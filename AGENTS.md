# Rules of Work (Marketing project: kitchen manufacturing services)

## 0) Goal (North Star)
Главная цель: вести проект **понятно, заботливо и предсказуемо** для русскоговорящего нетехнического специалиста.
Кодекс должен говорить “в моём мире”, а не в своём.

Приоритеты:
1) Минимум сущностей и файлов
2) Понятные названия (латиницей) и объяснения (по-русски)
3) Всегда актуальный `index.md` (карта проекта)
4) Всегда актуальный `glossary.md` (термины простыми словами)
5) Еженедельная гигиена: предлагать архивировать/удалять мусор (без самодеятельности)

---

## 1) Storage Model (где что живёт)
### 1.1 Local folder (Primary)
Локальная папка на компьютере — **основное рабочее пространство** для файлов и структуры проекта.

### 1.2 Obsidian (Human knowledge zone)
Obsidian — человеческая рабочая зона для заметок, решений и навигации.
Правило: синхронизация выполняется только в `kitchen_obsidian/` и только по категориям.
- Машинные сущности проекта в Obsidian автоматически не зеркалятся.
- Приоритет — ясные материалы для человека и помощь в принятии решений.

### 1.3 GitHub (Site only)
Версионируется только сайт (код/структура сайта).
Правила:
- В GitHub отправляется только содержимое папки сайта (см. структуру ниже).
- Тяжёлые медиа (аудио, большие видео) **в GitHub не коммитить**.

### 1.4 Two Zones (Machine vs Human)
Проект разделён на две зоны:
- **Machine zone**: `promotion/`, `analytics/`, `sales/`, `crm/`, `account-management/`, `project-management/`, `archive/`.
- **Human zone**: `kitchen_obsidian/` — человеко-ориентированные материалы для принятия решений и работы с терминами.

Правило: синхронизация применяется только к `kitchen_obsidian/`, machine zone не синхронизируется автоматически.

## 2) Golden Rules (обязательные правила)
1) **Никаких новых сущностей/подсущностей без моего явного одобрения.**
   - Если появляется идея новой сущности (например “email marketing”) — сначала спросить и дождаться “OK”.

2) **Перед созданием любых новых файлов/папок**:
   - сначала показать список: что создаёшь, где, зачем (1–2 строки на файл);
   - дождаться подтверждения.
   Исключение: мелкие правки существующих файлов (без создания новых).

3) **Никаких “служебных” папок и документов, которые понятны только Кодексу.**
   - Всё должно быть объяснимо человеку: зачем существует и как использовать.

4) **Имена файлов и папок только латиницей (ASCII), понятные, общепринятые.**
   - kebab-case или snake_case
   - без пробелов
   - без “final_final2”
   Примеры: `campaign-brief.md`, `plan-fact-2026-03-04.md`, `sales-call-review-2026-03-03.md`

5) **Каждый новый файл обязан иметь в начале:**
   - Title (заголовок)
   - Purpose (1–3 строки: зачем создан)
   - Owner (кто основной пользователь)
   - Last updated (дата)
   И после создания файл **должен быть добавлен в `index.md`**.

6) **Ничего не удалять без моего подтверждения.**
   - По умолчанию: предлагать “архивировать”, а не “удалять”.

7) **Ежедневный plan-fact обязателен** (см. `project-management/daily-plan-fact/`).

8) **Технические термины в документах для человека писать с расшифровкой.**
   - Для новых терминов обязательно добавлять простое объяснение и пример в `glossary.md`.

## 4) File Naming (правила именования)
### 4.1 General
- Только латиница (ASCII)
- `kebab-case` предпочтительнее
- Если документ привязан к дате (созвон, отчёт, план-факт) — добавлять дату:
  - `topic-YYYY-MM-DD.md`
- Если документ “артефакт” без дат (например, регламент CRM) — без версий:
  - `crm-architecture.md` (версионирование документов не требуется)

Примеры:
- `project-management/daily-plan-fact/plan-fact-2026-03-04.md`
- `promotion/yandex-direct/campaign-brief-2026-03-01.md`
- `crm/architecture/crm-architecture.md`
- `sales/manager-guides/sales-script.md`

### 4.2 Audio files
Аудио-файлы тоже латиницей:
- `call-sales-YYYY-MM-DD-clientname.mp3`
- `call-client-YYYY-MM-DD-topic.mp3`
Если есть Obsidian-копия — хранить путь/ID в реестре синхронизации.

## 5) index.md (обязательный “человеческий” справочник)
`index.md` — карта проекта для нетехнического специалиста.

### 5.1 What index.md must contain
1) **Project status** (5–10 строк): что делаем сейчас, что важно, ближайшие шаги
2) По каждой сущности — коротко “зачем она”
3) По каждой сущности — список файлов и понятное объяснение

### 5.2 Required format for each file entry
Для каждого файла (или крупной папки) обязательно:
- **Path**: `promotion/yandex-direct/...`
- **Purpose (RU)**: зачем создан простыми словами
- **Trigger**: почему появился (созвон / запуск кампании / просьба клиента / аудит и т.д.)
- **Owner**: кто главный пользователь (вы / клиент / команда)
- **Last updated**: дата
- **Lifecycle**: `Active` или `Archived`
- **Next step** (если нужно): что с этим делать дальше

Правило: создал новый файл → сразу добавил запись в `index.md`.

## 6) glossary.md (обязательный глоссарий)
`glossary.md` — словарь терминов для русскоговорящего нетехнического специалиста.

### 6.1 Rules
- Любой новый термин/аббревиатура из маркетинга/продаж/CRM добавляется в `glossary.md`.
- Писать простыми словами: “что это”, “зачем нам”, “пример для кухонь”, “частые ошибки”.

### 6.2 Entry format
- Термин (RU/EN)
- Простое определение (1–3 предложения)
- Зачем важно в этом проекте
- Пример на кейсе “кухни”
- Частая ошибка/миф (если есть)

---

## 7) Daily Plan-Fact (строго каждый день)
Файл создаётся ежедневно:
`project-management/daily-plan-fact/plan-fact-YYYY-MM-DD.md`

Структура файла:
1) Plan (что сделаем сегодня) — 3–10 пунктов
2) Fact (что реально сделано) — 3–10 пунктов
3) Metrics (если есть) — лиды/бюджет/заявки/CR/CTR кратко
4) Blockers (что мешало)
5) Next day focus (главное на завтра)

Кодекс не должен пропускать день: если файла нет — предложить создать.

---

## 8) Archiving & Cleanup (гигиена проекта)
Раз в неделю (ориентир: конец недели) Кодекс должен предложить:
- список файлов/папок, которые давно не обновлялись или потеряли актуальность;
- объяснение простыми словами “почему можно убрать”;
- действие по умолчанию: **архивировать** в `archive/` (а не удалять).

Правила:
- Ничего не переносить/не удалять без моего подтверждения.
- При архивации — обновить `index.md`: `Lifecycle = Archived` + причина.

---

## 9) CRM Access & Sensitive Data
Ограничений по персональным данным нет, но:
- В `crm/access/` желательно хранить **не сами пароли**, а:
  - кто и какой доступ должен иметь,
  - где хранится пароль/ключ (если есть),
  - инструкция “как запросить/восстановить доступ”.
- Если я явно даю секрет — можно хранить, но всё равно предпочитать “аккуратный формат”.

---

## 10) How to act on user requests (standard workflow)
1) Сформулировать задачу человечески (1–2 строки).
2) Предложить план действий и **перечень файлов, которые будут созданы/обновлены**.
3) Дождаться подтверждения, если создаются новые файлы/папки.
4) Выполнить работу минимально необходимым количеством файлов.
5) Обновить `index.md` и при необходимости `glossary.md`.

## 12) Obsidian Sync Policy (двусторонняя синхронизация)
Цель: синхронизировать только `kitchen_obsidian/vault/` ↔ `kitchen_obsidian/local-zone/` без потери данных.

Правила:
1) Направление: **двустороннее** (`vault -> local-zone` и `local-zone -> vault`).
2) Частота: каждые 15 минут (`launchd`, `StartInterval=900`).
3) Конфликты: побеждает более новый timestamp (если одинаково — приоритет у `vault`).
4) Удаление запрещено: вместо удаления — архивирование с причиной.
5) `kitchen_obsidian/registry/*` не синхронизируется как контент заметок.
6) Секреты/API ключи не хранить в репозитории.
