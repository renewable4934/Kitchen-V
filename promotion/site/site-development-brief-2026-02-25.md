# ТЗ: Два сайта (Кухни и Шкафы) на Next.js + Vercel + v0 + Codex/Zed

Актуально на: **25 февраля 2026**

## 1) Цель документа

Зафиксировать единое техническое задание на запуск двух коммерческих сайтов:

1. Сайт по продаже кухонь на заказ.
2. Сайт по продаже шкафов на заказ.

Дополнительно: построить повторяемый процесс разработки сайтов с помощью нейросетей, чтобы новые сайты запускались быстрее и с предсказуемым качеством.

## 2) Бизнес-цели и KPI

Целевые метрики для каждого сайта (MVP-период 30 дней после запуска):

1. Конверсия сайта в заявку: `>= 4.5%`.
2. Стоимость лида (CPL): в рамках целевого бюджета из операционного плана.
3. SLA первого ответа по заявке: `<= 5 минут`.
4. Доля валидных лидов (корректный телефон + целевой запрос): `>= 60%`.

## 3) Границы проекта

Входит в текущий этап:

1. Проектирование и запуск 2 сайтов (Кухни / Шкафы).
2. Настройка CI/CD и деплоя на Vercel.
3. Базовая аналитика (события, отправка лидов, UTM).
4. Документированный AI-процесс: `v0 -> Codex -> Zed -> Vercel`.

Не входит в текущий этап:

1. Полноценная CRM-автоматизация сделок (кроме передачи лида).
2. Сложный личный кабинет клиента.
3. Многоязычность.

## 4) Технологический стек и роли инструментов

1. `Next.js` (App Router, TypeScript): основной фронтенд и API-роуты для формы.
2. `Vercel`: preview-окружения, production-деплой, переменные окружения.
3. `v0.app`: генерация UI-секций и прототипов компонентов.
4. `Codex + Zed`: ускорение разработки, рефакторинг, проверка структуры, правка кода.

Ограничение: `v0` используется для UI и прототипирования, но интеграции (аналитика, API, формы, валидация) реализуются вручную в коде.

## 5) Архитектурный подход (обязательный)

Использовать **template-first** подход:

1. Один базовый шаблон сайта (дизайн-система, формы, аналитика, SEO-блоки).
2. Два отдельных проекта на его основе:
   - `kitchen-site`
   - `wardrobe-site`
3. В каждом проекте единая точка настройки контента: `site.config.ts` (бренд, оффер, контакты, кейсы, FAQ).

Зачем: минимизируем ручные отличия и получаем систему, из которой можно быстро масштабировать новые сайты.

## 6) Функциональные требования по сайтам

### 6.1 Общие для обоих сайтов

1. Главная страница с оффером, CTA и формой заявки.
2. Блок преимуществ.
3. Блок портфолио (кейсы/фото с описанием).
4. Блок этапов работы (замер -> проект -> производство -> монтаж).
5. Блок цен/диапазонов стоимости.
6. FAQ.
7. Контакты + карта/адрес + кликабельные способы связи.
8. Страница политики конфиденциальности.
9. Форма заявки с валидацией телефона.
10. Передача UTM-меток и `funnel_type` в лид.

### 6.2 Сайт "Кухни"

1. УТП с акцентом на эргономику и материалы фасадов.
2. Сегменты: угловые, прямые, П-образные, кухни-гостиные.
3. Квиз: стиль, размер, бюджет, срок изготовления.
4. Отдельный оффер на бесплатный замер/дизайн-проект (если подтвержден бизнесом).

### 6.3 Сайт "Шкафы"

1. УТП с акцентом на хранение и экономию пространства.
2. Сегменты: купе, распашные, гардеробные, встроенные решения.
3. Квиз: тип шкафа, размеры, наполнение, бюджет.
4. Оффер на расчет стоимости по размерам.

## 7) Технические требования

1. `TypeScript` включен для всего нового кода.
2. Компонентный подход: переиспользуемые секции в `components/`.
3. Server Components по умолчанию, Client Components только где реально нужен интерактив.
4. Обработка формы через `Next.js Route Handlers` (`/api/lead`).
5. Валидация входных данных на сервере (например, `zod`).
6. Защита от спама: honeypot + rate limit (минимум на уровне IP).
7. Все секреты только через переменные окружения Vercel.
8. Поддержка preview deploy на каждую ветку.

## 8) Интеграции и данные

1. Интеграция с CRM webhook:
   - приоритет: `BITRIX24_WEBHOOK_URL`
   - fallback: `CRM_WEBHOOK_URL`
2. Fallback-хранилище лидов при недоступной CRM (лог/хранилище).
3. Отправка ключевых событий:
   - `view_offer`
   - `start_quiz`
   - `submit_lead`
   - `click_call`
   - `open_whatsapp`
4. UTM-параметры сохраняются и передаются вместе с лидом.

## 9) SEO и производительность

1. Уникальные `title`, `description`, `h1` для каждого сайта.
2. Schema.org (`LocalBusiness`, `FAQPage`, при необходимости `Product`).
3. ЧПУ URL, корректные canonical.
4. Open Graph и базовые соц-мета-теги.
5. Целевые показатели (mobile):
   - Lighthouse Performance: `>= 80`
   - Accessibility: `>= 90`
   - Best Practices: `>= 90`
   - SEO: `>= 90`

## 10) UX/UI требования

1. Адаптивность от `320px` до desktop.
2. CTA-кнопки в первом экране и после ключевых блоков.
3. Форма заявки доступна без длинной прокрутки.
4. Не использовать тяжелые визуальные эффекты, мешающие скорости.
5. Единый стиль, но разная визуальная идентика для кухонь и шкафов.

## 11) AI-процесс (системный конвейер)

### Шаг 1. Бриф

1. Заполнить шаблон: ЦА, офферы, диапазоны цен, гео, контакты, FAQ.
2. Зафиксировать это в `site.config.ts`.

### Шаг 2. Прототип в v0

1. Сгенерировать 2-3 варианта hero + секций.
2. Выбрать вариант по критериям: читаемость оффера, плотность CTA, мобильность.
3. Экспортировать компоненты в проект.

### Шаг 3. Интеграция через Codex + Zed

1. Codex: встраивание компонентов, API-форма, аналитика, SEO.
2. Zed: ручная проверка кода, быстрая правка и навигация.
3. Проверка структуры и нейминга перед коммитом.

### Шаг 4. Тесты и QA

1. Smoke-тесты: открытие страниц, отправка формы, корректность событий.
2. Проверка mobile/desktop.
3. Проверка preview-сборки на Vercel.

### Шаг 5. Релиз

1. Production deploy.
2. Проверка метрик в первые 24 часа (ошибки, лиды, событие `submit_lead`).

## 12) План обучения (с нуля)

### Неделя 1: База Next.js

1. Понять App Router, layout, page, client/server компоненты.
2. Научиться делать форму и API route.
3. Собрать простую страницу с 3-4 секциями.

### Неделя 2: Vercel и окружения

1. Подключить Git-репозиторий к Vercel.
2. Настроить preview + production.
3. Настроить env vars и проверить деплой без локальных костылей.

### Неделя 3: v0 + интеграция

1. Генерировать UI в v0 по шаблонному промпту.
2. Переносить код в проект и приводить к единой дизайн-системе.
3. Убирать лишнее и стандартизировать компоненты.

### Неделя 4: Codex + Zed рабочий цикл

1. Превратить повторяемые задачи в чеклист (форма, SEO, аналитика, QA).
2. Зафиксировать prompt-шаблоны для типовых задач.
3. Сделать 1 полный цикл запуска (от брифа до релиза) за 1-2 дня.

## 13) Критерии готовности (Definition of Done)

Сайт считается готовым, если:

1. Развернут на Vercel в production.
2. Все обязательные страницы и секции из раздела 6 реализованы.
3. Форма отправляет лид с `funnel_type` и UTM.
4. Аналитические события фиксируются корректно.
5. Lighthouse и мобильная адаптивность соответствуют разделу 9.
6. Есть короткая эксплуатационная инструкция в `README` проекта сайта.

## 14) Риски и меры

1. Риск: слабая экспертиза в стеке.
   - Мера: фиксированный 4-недельный план обучения и шаблоны задач.
2. Риск: нестабильное качество AI-генерации UI.
   - Мера: принимать компоненты только после ручного QA и унификации.
3. Риск: потеря лидов из-за ошибок формы/CRM.
   - Мера: fallback-лог лидов + алерты на ошибки отправки.

## 15) Артефакты по итогу этапа

1. Два production-сайта: Кухни и Шкафы.
2. Репозиторный шаблон для следующего сайта.
3. Набор prompt-шаблонов для v0 и Codex.
4. Чеклист запуска (QA + SEO + Analytics + CRM).
